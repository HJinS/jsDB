# Key Encoding Story

B+Tree의 key encoding/decoding이 왜 현재 구조가 됐는지에 대한 설계 결정 기록.
각 변경은 특정 문제에서 출발했고, 최종 목표는 **스키마 없이 바이트 비교(memcmp)만으로 key 대소를 판단할 수 있는 포맷**이다.

---

## 문제 1 — 부호 있는 정수를 raw bytes로 저장하면 비교가 틀림

**관련 커밋:** `2883408` (BUG-004)

two's complement 정수는 음수의 최상위 비트(MSB)가 1이다.
그래서 raw bytes를 그대로 비교하면 음수가 양수보다 항상 크게 나온다.

```
-1 → 11111111 11111111 11111111 11111111
 0 → 00000000 00000000 00000000 00000000
 1 → 00000000 00000000 00000000 00000001

바이트 비교: 11111111... > 00000000... → -1 > 0  (틀림)
```

**해결: MSB만 뒤집기 (XOR MIN_VALUE)**

부호 비트만 뒤집으면 음수와 양수의 순서가 바로잡힌다.

```
MIN_VALUE = 10000000 00000000 00000000 00000000

-1 XOR MIN_VALUE = 01111111 11111111 11111111 11111111
 0 XOR MIN_VALUE = 10000000 00000000 00000000 00000000
 1 XOR MIN_VALUE = 10000000 00000000 00000000 00000001

바이트 비교: 01111111... < 10000000... < 10000000...00000001
            → -1 < 0 < 1  (맞음)
```

Float/Double은 IEEE 754 구조라 별도 처리가 필요하다.

```
// IEEE 754 float: [부호 1비트][지수 8비트][가수 23비트]

양수 1.0  = 0 01111111 00000000000000000000000
양수 2.0  = 0 10000000 00000000000000000000000
음수 -1.0 = 1 01111111 00000000000000000000000
음수 -2.0 = 1 10000000 00000000000000000000000

// 양수: MSB만 뒤집으면 충분
1.0 XOR 10000000... = 1 01111111 00000000000000000000000
2.0 XOR 10000000... = 1 10000000 00000000000000000000000
→ 바이트 비교 시 1.0 < 2.0  (맞음)

// 음수: 전체 비트 반전 필요
// 이유: 음수는 지수가 클수록 절댓값이 커지는데,
//       MSB만 뒤집으면 -2.0 > -1.0이 돼버림
-1.0 inv() = 0 10000000 11111111111111111111111
-2.0 inv() = 0 01111111 11111111111111111111111
→ 바이트 비교 시 -2.0 < -1.0  (맞음)
```

## 문제 2 — LITTLE_ENDIAN으로 저장하면 멀티바이트 정수 비교가 틀림

**관련 커밋:** `452d3b0`

멀티바이트 정수는 높은 자리 바이트가 앞에 와야(BIG_ENDIAN) 바이트 비교가 올바른 대소 관계를 만든다.

```
256 (LITTLE_ENDIAN): [00000000][00000001]  ← 낮은 바이트부터
  1 (LITTLE_ENDIAN): [00000001][00000000]
→ 바이트 비교: 00000000 < 00000001 → 256 < 1  (틀림)

256 (BIG_ENDIAN): [00000001][00000000]  ← 높은 바이트부터
  1 (BIG_ENDIAN): [00000000][00000001]
→ 바이트 비교: 00000001 > 00000000 → 256 > 1  (맞음)
```

JVM `ByteBuffer`의 기본값이 BIG_ENDIAN이라 기능적으론 변화 없으나, 의도를 명시적으로 고정했다.

## 문제 3 — String length-prefix 방식이 사전식 비교를 깸

**관련 커밋:** `987ef08`

길이를 앞에 붙이면 바이트 비교 시 길이 숫자가 먼저 비교된다.

```
"abc" → [00000011][01100001][01100010][01100011]
         길이=3      'a'        'b'        'c'

"b"   → [00000001][01100010]
         길이=1      'b'

바이트 비교: 00000011 > 00000001 → "abc" > "b"  (틀림, 사전식으론 "b" > "abc")
```

**해결: null terminator 방식으로 변경**

끝에 `00000000`을 붙이면 문자열 내용 자체가 비교된다.

```
"abc" → [01100001][01100010][01100011][00000000]
          'a'        'b'        'c'     terminator

"b"   → [01100010][00000000]
          'b'       terminator

바이트 비교: 01100001 < 01100010 → "abc" < "b"  (맞음)
```

문자열 내부에 `00000000`이 있는 경우 terminator로 오인하지 않도록 `00000000 11111111`로 escape한다.

```
"a\0b" → [01100001][00000000][11111111][01100010][00000000]
           'a'        escape 시작    escape 끝  'b'    terminator
```

## 문제 4 — 고정폭 타입의 00000000이 terminator로 오인됨

**관련 커밋:** `635b528`

문제 1에서 `encodeSortable()`을 도입한 뒤, INT 등 고정폭 타입의 결과 바이트에도 `00000000`이 생길 수 있다.
이 값을 그대로 저장하면 decode 시 `00000000`을 terminator로 오인해 잘라버린다.

```
INT 256 XOR MIN_VALUE
= 00000000 00000001 XOR 10000000 00000000
= 10000000 00000001

// 4바이트로 표현
= 10000000 00000000 00000001 00000000
           ↑ 여기서 terminator로 오인 → 잘림
```

**해결: 고정폭 타입도 동일하게 escape 적용**

모든 타입이 escape + terminator 포맷으로 통일된다.

```
변환 후 bytes: [10000000][00000000][00000001][00000000]

escape 적용:   [10000000][00000000][11111111][00000001][00000000][11111111][00000000]
                          └─ escape ─┘                  └─ escape ─┘         terminator
```

decode 측도 모든 타입을 terminator 스캔(`readVarType`)으로 통일했다.

## escape가 memcmp에 영향을 끼치지 않는 이유

escape는 `00000000`을 `00000000 11111111`로 치환하는데, 이것이 바이트 비교 순서를 깨지 않는 이유는
**`00000000`이 가능한 모든 바이트 값 중 가장 작기 때문**이다.

두 값을 비교할 때 처음으로 달라지는 위치 i에서:

**한쪽만 `00000000`일 때:**
```
A: [... 00000000 11111111 ...]  (data 00000000, escaped)
B: [... 00000001 ...]           (data 00000001)

byte i: 00000000 vs 00000001 → A < B  (맞음, 원본도 A[i]=0 < B[i]=1)
```

`00000000`이 가장 작은 값이므로 escape 마커가 오든 terminator가 오든 결과가 같다.

**둘 다 `00000000`일 때:**
```
A: [... 00000000 11111111 00000001 ...]
B: [... 00000000 11111111 00000010 ...]

byte i  : 00000000 == 00000000  (둘 다 escape 시작)
byte i+1: 11111111 == 11111111  (둘 다 escape 마커)
byte i+2: 00000001 vs 00000010 → A < B  (맞음, 원본도 A[i+1]<B[i+1])
```

둘 다 같은 위치에서 escape됐으므로 `00000000 11111111`이 상쇄되고, 그 다음 바이트로 비교가 넘어간다.

escape marker를 `11111111`(가장 큰 값)로 선택한 이유도 여기 있다.
두 번째 바이트는 비교에 영향을 줄 일이 없지만, 혹시 영향을 줄 경우에도 `11111111`은 어떤 값보다도 크거나 같아서 순서를 역전시킬 수 없다.

## 최종 포맷 요약

```
모든 컬럼 타입의 encode 결과:

  [null flag: 0x00=null, 0x01=not null] + [값 bytes (내부 0x00은 escape)] + [terminator 0x00]

descending 컬럼: 위 결과 전체를 invert() (모든 비트 반전)
```

이로써 복합 key 전체를 스키마 없이 바이트 하나씩 순서대로 비교하는 것만으로 대소 판단이 가능해졌다.
